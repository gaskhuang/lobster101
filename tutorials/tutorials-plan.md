## 教案 01：每日情報摘要

### 標題
15 分鐘打造你的 AI 每日情報摘要

### 副標
讓 AI 每天早上 8 點幫你讀完 50 則新聞，產出一頁精華摘要——你只需要喝咖啡。

### 適合誰
- 每天花超過 1 小時瀏覽產業新聞的創業者、行銷主管、顧問
- 想掌握市場動態但時間有限的中小企業老闆
- 需要定期彙報產業趨勢的研究人員或幕僚

### 解決什麼問題
手動追蹤多個新聞來源（RSS、電子報、社群）既耗時又容易遺漏關鍵資訊。你需要一個每日自動幫你彙整、摘要、分類的流程，省下每天 1–2 小時的閱讀時間。

### 所需工具
- OpenClaw CLI
- Search & Research 類 Skills（RSS Reader、Web Scraper）
- AI Summarization Skill
- Telegram / Email 推送 Skill
- 排程工具（cron 或 n8n）

### 難度
⭐⭐ 入門（有基本終端機操作經驗即可）

### 預估時間
15–20 分鐘（首次設定）

### 步驟

**步驟 1：定義你的情報來源**
列出 5–10 個你每天必看的資訊來源：產業新聞網站、競品 Blog、Twitter/X 帳號、Reddit 子版。把它們整理成一張清單，包含 RSS 連結或網址。

**步驟 2：安裝所需 Skills**
```bash
npx clawhub@latest install rss-reader
npx clawhub@latest install web-scraper
npx clawhub@latest install ai-summarizer
npx clawhub@latest install telegram-notifier
```

**步驟 3：設定 RSS 抓取任務**
建立一個 `sources.yaml` 設定檔，將你的來源 URL 逐一填入。設定抓取頻率為每日一次，時間設在早上 7:30。

**步驟 4：配置摘要模板**
撰寫 Prompt 模板，告訴 AI 你要的摘要格式：「列出前 10 則最重要新聞，每則用一句話摘要，附上原始連結，最後給出 3 個行動建議。」

**步驟 5：設定分類邏輯**
依照你關心的面向（競品動態、法規變化、技術趨勢、市場數據）設定自動分類標籤，讓摘要報告一目了然。

**步驟 6：串接推送管道**
將產出的摘要接上 Telegram Bot 或 Email，讓報告每天早上 8 點自動送到你手上。

**步驟 7：設定排程自動化**
用 cron job 或 n8n 將整條流程串起來，設定每日定時觸發。確認排程正確執行。

**步驟 8：測試與微調**
手動觸發一次，檢查摘要品質。調整 Prompt 中的摘要長度、分類標籤、推送格式，直到符合你的閱讀習慣。

### 常見錯誤
1. **來源太多太雜**：一開始不要超過 10 個來源，否則摘要會又長又沒重點。先聚焦核心來源，再逐步擴充。
2. **Prompt 太模糊**：「幫我摘要」不夠好，要明確指定格式、長度、分類方式。
3. **忽略 RSS 格式差異**：不同網站的 RSS 結構不同，部分網站可能需要用 Web Scraper 取代 RSS Reader。
4. **沒設錯誤通知**：排程失敗時你不會知道。務必加上失敗通知機制。

### 升級方向
- 加入競品定價監控：自動偵測競品價格或功能變動
- 串接 Notion / Google Sheet：將每日摘要自動歸檔，累積成可搜尋的知識庫
- 多語言情報：抓取英文、日文來源並自動翻譯成中文
- 週報自動產出：每週五自動彙整當週摘要，產出一份週報

## 教案 02：語音筆記轉任務

### 標題
用 AI 把語音筆記自動變成待辦清單

### 副標
開車、走路、開會時隨口說，AI 幫你轉文字、拆任務、排進待辦——從此不再忘記任何靈感。

### 適合誰
- 常在移動中產生想法但來不及記錄的業務、顧問、創業者
- 開會後總忘記 action items 的團隊負責人
- 習慣用語音備忘但事後懶得整理的自由工作者

### 解決什麼問題
語音筆記通常錄完就被遺忘。手動聽錄音、抄筆記、拆任務太花時間。你需要一條自動化流程：錄音 → 轉文字 → 抽取任務 → 寫入待辦工具。

### 所需工具
- OpenClaw CLI
- Speech-to-Text Skill（Whisper 本地端或雲端 API）
- AI Task Extraction Skill
- Todoist / Notion / Trello 整合 Skill
- 手機錄音 App（任何可匯出音檔的 App）

### 難度
⭐⭐ 入門

### 預估時間
20–25 分鐘

### 步驟

**步驟 1：選擇錄音工具**
在手機上選一個可以自動同步錄音檔的 App。iOS 用「語音備忘錄」搭配 iCloud，Android 用「Google 錄音工具」搭配 Google Drive。確保錄音檔能自動同步到電腦或雲端。

**步驟 2：安裝語音轉文字 Skill**
```bash
npx clawhub@latest install whisper-transcriber
npx clawhub@latest install task-extractor
npx clawhub@latest install todoist-connector
```

**步驟 3：設定監聽資料夾**
指定一個資料夾（如 `~/VoiceNotes/`），讓 Agent 自動偵測新增的音檔。一有新錄音就觸發轉錄流程。

**步驟 4：配置轉錄參數**
設定語言（繁體中文）、模型大小（建議 medium 以上以確保中文準確度）、輸出格式（純文字 + 時間戳記）。

**步驟 5：撰寫任務抽取 Prompt**
告訴 AI：「從以下逐字稿中，抽取所有可執行的任務。每個任務需包含：任務描述、優先度（高/中/低）、建議截止日期。忽略閒聊和重複內容。」

**步驟 6：串接待辦工具**
將抽取出的任務自動寫入你常用的工具：Todoist、Notion Database、或 Trello 看板。設定對應的欄位映射。

**步驟 7：加入確認機制**
在自動寫入前，先將任務清單推送到 Telegram 讓你確認。你回覆「OK」後才正式寫入待辦工具，避免 AI 誤判。

**步驟 8：建立日常習慣**
設定每天固定時段（如晚上 9 點）自動處理當天所有語音筆記，隔天早上你就能看到整理好的任務清單。

### 常見錯誤
1. **錄音品質太差**：在嘈雜環境錄音會嚴重影響轉錄準確度。盡量在安靜處錄，或使用降噪麥克風。
2. **語音太隨性**：完全無結構的意識流很難抽取任務。錄音時稍微有意識地說「任務一⋯⋯任務二⋯⋯」會大幅提升效果。
3. **一次處理太多檔案**：長音檔（超過 30 分鐘）建議先切段再處理，避免記憶體不足或摘要品質下降。
4. **沒有人工確認就直接寫入**：AI 可能誤判優先度或截止日，務必保留確認步驟。

### 升級方向
- 會議錄音自動分派：辨識不同發言人，將任務自動分派給對應的人
- 多語言支援：英文會議也能轉錄並抽取中文任務
- 語音日記：除了任務，也自動整理成每日工作日誌
- 串接行事曆：有截止日的任務自動建立日曆事件

## 教案 03：Telegram 客服自動回覆

### 標題
打造 24/7 Telegram 客服自動回覆機器人

### 副標
客戶深夜發問也能秒回——用 AI 處理 80% 重複問題，真人只需處理剩下 20%。

### 適合誰
- 每天花超過 1 小時回覆客戶重複問題的電商賣家、線上課程經營者
- 客服人力不足但又不想讓客戶等太久的中小企業
- 想在 Telegram 群組或頻道提供自動問答的社群經營者

### 解決什麼問題
客戶的問題有 80% 是重複的（價格、退貨、使用方式、營業時間）。手動回覆浪費人力、回覆慢導致客戶流失。你需要一個能理解問題、查找知識庫、自動回覆的 AI 客服 Bot。

### 所需工具
- OpenClaw CLI
- Telegram Bot Skill
- RAG（檢索增強生成）Skill
- Knowledge Base Skill（FAQ 知識庫）
- 人工轉接 Skill（處理 AI 無法回答的問題）

### 難度
⭐⭐⭐ 中等（需要建立知識庫 + 設定 Bot）

### 預估時間
40–60 分鐘

### 步驟

**步驟 1：建立 Telegram Bot**
在 Telegram 找 @BotFather，輸入 `/newbot` 建立一個新的 Bot。取得 Bot Token 並妥善保存。設定 Bot 的名稱與描述。

**步驟 2：整理你的 FAQ 知識庫**
把客戶最常問的 30–50 個問題整理成結構化文件（Markdown 或 JSON）。每個問答對包含：問題、標準回答、相關連結。

**步驟 3：安裝相關 Skills**
```bash
npx clawhub@latest install telegram-bot
npx clawhub@latest install rag-engine
npx clawhub@latest install knowledge-base
```

**步驟 4：建立 RAG 知識庫索引**
將你的 FAQ 文件餵入 RAG Engine，建立向量索引。這讓 AI 能夠根據客戶的「自然語言提問」找到最相關的答案，而不只是關鍵字比對。

**步驟 5：設定回覆邏輯**
配置 AI 的回覆策略：
- 信心度 > 85%：直接回覆答案
- 信心度 60–85%：回覆答案但附上「如需進一步協助，請輸入 /human」
- 信心度 < 60%：自動轉接真人客服

**步驟 6：設計回覆模板**
為不同情境設計回覆語氣與格式。例如：詢價用正式語氣 + 附價目表連結；售後問題用親切語氣 + 附操作教學連結。

**步驟 7：串接人工轉接**
當 AI 無法回答或客戶主動要求時，自動通知真人客服接手。設定通知管道（另一個 Telegram 群組或 Email）。

**步驟 8：上線測試與監控**
先在內部測試群組模擬 20 種不同問法，確認回覆品質。上線後持續監控「未命中」的問題，每週更新知識庫。

### 常見錯誤
1. **知識庫太少**：只放 10 個 FAQ 不夠。客戶問法多變，至少要 30 個以上的問答對，且每個問題要有 2–3 種不同問法。
2. **沒設轉接機制**：AI 回答不了卻沒通知真人，客戶會感覺被忽略，反而更差。
3. **語氣太機器人**：回覆模板要有品牌語氣，不要像制式回覆。加入 emoji、稱呼客戶名字。
4. **忽略持續優化**：上線後第一週一定會有大量 AI 答不好的問題。每天花 10 分鐘看未命中紀錄，持續更新知識庫。

### 升級方向
- 多語系客服：自動偵測客戶語言，用對應語言回覆
- 訂單查詢整合：串接你的訂單系統，客戶可以直接用 Bot 查物流狀態
- 主動推播：定期推送促銷、新品資訊給曾互動過的客戶
- 數據分析儀表板：統計最常被問的問題、回覆滿意度、轉接率

## 教案 04：第二大腦 / 個人知識庫

### 標題
建立你的 AI 第二大腦——再也不用 Google 同一件事兩次

### 副標
把散落在 Notion、Google Drive、書籤、聊天紀錄裡的知識統統收進來，用自然語言隨問隨查。

### 適合誰
- 資料散落在 5 個以上工具裡、每次要找東西都找半天的知識工作者
- 讀過很多書和文章但總是記不住重點的學習者
- 需要快速調閱過往研究、筆記、會議紀錄的顧問或研究員

### 解決什麼問題
你的知識分散在太多地方：Notion 筆記、Google Docs、瀏覽器書籤、PDF 文件、Slack 對話。傳統搜尋只能比對關鍵字，找不到「語意相關」的內容。你需要一個統一的知識庫，能用自然語言問問題並得到精準答案。

### 所需工具
- OpenClaw CLI
- RAG Engine Skill
- Document Loader Skills（支援 Markdown、PDF、Notion、Google Docs）
- Embedding Skill（向量化）
- CLI 或 Web 查詢介面

### 難度
⭐⭐⭐ 中等

### 預估時間
30–45 分鐘（基礎版）

### 步驟

**步驟 1：盤點你的知識來源**
列出你目前所有的知識存放位置：Notion workspace、Google Drive 資料夾、本地 Markdown 檔案、常用網站書籤、PDF 電子書。按優先度排序。

**步驟 2：安裝知識庫 Skills**
```bash
npx clawhub@latest install rag-engine
npx clawhub@latest install doc-loader
npx clawhub@latest install notion-connector
npx clawhub@latest install pdf-parser
```

**步驟 3：設定資料來源連接**
逐一設定各資料來源的 API 金鑰或存取路徑。Notion 需要 Integration Token，Google Drive 需要 Service Account。本地檔案直接指定資料夾路徑。

**步驟 4：執行首次資料匯入**
啟動資料匯入流程。系統會讀取所有文件、切段、向量化，存入本地向量資料庫。首次匯入視資料量可能需要 5–30 分鐘。

**步驟 5：設定增量同步**
配置定時同步（每日或即時），讓新增或修改的文件自動更新到知識庫。避免知識庫和原始資料脫節。

**步驟 6：測試查詢效果**
用自然語言問幾個你知道答案的問題，測試查詢準確度。例如：「上次和客戶 A 開會討論了什麼？」「我們的退貨政策是什麼？」

**步驟 7：微調檢索參數**
根據測試結果調整：切段大小（chunk size）、相似度閾值、回傳數量（top-k）。通常 chunk size 500–1000 字元、top-k 3–5 是好的起點。

**步驟 8：建立查詢快捷方式**
設定 CLI alias 或 Telegram Bot 介面，讓你能隨時快速查詢。例如：`brain "去年Q4營收報告重點"` 就能得到答案。

### 常見錯誤
1. **一次匯入太多低品質資料**：先從最重要、最常用的 100 份文件開始。垃圾進、垃圾出——低品質文件會污染搜尋結果。
2. **chunk size 設太大或太小**：太大（>2000 字）會包含太多無關內容；太小（<200 字）會喪失上下文。
3. **忘記定期更新**：知識庫建好後就忘了更新，三個月後資料就過時了。務必設定自動同步。
4. **只匯入不整理**：匯入前先清理資料來源：刪除過時筆記、合併重複文件、統一命名規則。

### 升級方向
- 團隊知識庫：讓整個團隊共享同一個知識庫，搭配權限控制
- 自動筆記整理：新增的筆記自動分類、打標籤、建立關聯
- 智慧寫作助手：寫文章時自動從知識庫中拉出相關資料作為參考
- 語音查詢：用語音提問，AI 語音回答（適合開車或做家事時使用）

## 教案 05：自動會議紀錄與行動項目

### 標題
開完會自動產出會議紀錄與行動項目

### 副標
不用再指派「會議記錄員」——AI 幫你聽、幫你記、幫你追蹤，每個 action item 都不漏。

### 適合誰
- 每週開超過 5 場會議、會後紀錄總是拖延或遺漏的主管
- 會議中忙著討論沒空記筆記的團隊成員
- 需要追蹤跨部門 action items 的 PM 或專案經理

### 解決什麼問題
會議開完大家就忘了講什麼、該做什麼。手動記會議紀錄費時又不完整。Action items 沒人追蹤，下次開會又在討論同樣的事。你需要自動化的會議紀錄 + 行動追蹤系統。

### 所需工具
- OpenClaw CLI
- Speech-to-Text Skill（Whisper / Google STT）
- Meeting Summarizer Skill
- Task Extraction Skill
- Notion / Google Docs 整合 Skill
- 會議錄音工具（OBS、手機、或會議軟體內建錄音）

### 難度
⭐⭐⭐ 中等

### 預估時間
30–40 分鐘

### 步驟

**步驟 1：選擇錄音方式**
線上會議：開啟 Google Meet / Zoom 內建錄音功能，或用 OBS 錄製系統音訊。實體會議：用手機或錄音筆放在桌上。確保錄音品質清楚。

**步驟 2：安裝會議紀錄 Skills**
```bash
npx clawhub@latest install whisper-transcriber
npx clawhub@latest install meeting-summarizer
npx clawhub@latest install task-extractor
npx clawhub@latest install notion-connector
```

**步驟 3：設定逐字稿轉錄**
配置語言（中文/英文/混合）、模型大小、說話人辨識（diarization）。說話人辨識能區分「誰說了什麼」，對會議紀錄至關重要。

**步驟 4：設計會議紀錄模板**
定義你要的紀錄格式：
- 會議基本資訊（日期、參與者、時長）
- 討論重點（按議題分段）
- 決議事項
- 行動項目（負責人、截止日、狀態）

**步驟 5：配置行動項目抽取**
撰寫 Prompt：「從會議紀錄中抽取所有行動項目。每項包含：任務描述、負責人（根據逐字稿中的發言判斷）、建議截止日、優先度。」

**步驟 6：串接筆記工具**

**步驟 7：建立追蹤機制**
設定每日提醒：在 Telegram 或 Slack 推送即將到期的 action items。每週五自動產出「本週未完成項目清單」。

**步驟 8：團隊上線與回饋**
讓團隊試用 2 週，收集回饋。常見需調整的地方：紀錄格式、行動項目的粒度、提醒頻率。持續迭代。

### 常見錯誤
1. **錄音品質差**：線上會議如果收音不好，轉錄準確度會暴跌。建議參與者用耳機麥克風，減少迴音。
2. **沒做說話人辨識**：不知道誰說了什麼，行動項目就無法正確分派。這步不能省。

3. **忽略正面提及**：正面口碑是最好的行銷素材。主動回應、分享、甚至邀請客戶成為品牌大使。
4. **沒有長期趨勢分析**：只看即時提及不夠。要觀察每週、每月的情緒趨勢，才能發現系統性問題。

### 升級方向
- 自動回應：AI 根據回應 SOP 自動在平台上回覆（需謹慎設定）
- 競品深度分析：不只監控提及，還分析競品的行銷策略、新品發布、定價變動

## 教案 10：YouTube 摘要與 Podcast 流水線

### 標題
YouTube 影片秒變文字摘要——打造你的 Podcast 內容流水線

### 副標
看完一支 1 小時影片只需要 3 分鐘——AI 幫你轉錄、摘要、抽重點，還能自動產出 Podcast 逐字稿。

### 適合誰
- 每天要看大量 YouTube 教學或產業影片但時間不夠的學習者
- 想把 YouTube 內容重新包裝成 Podcast 或文章的內容創作者
- 需要快速掌握影片重點來做決策的主管

### 解決什麼問題
YouTube 影片動輒 30–60 分鐘，全部看完太花時間。你只想知道重點。如果你是內容創作者，你可能想把影片內容轉成 Podcast 或文字形式，但手動轉錄和編輯太累。

### 所需工具
- OpenClaw CLI
- YouTube Downloader Skill
- Speech-to-Text Skill（Whisper）
- Video Summarizer Skill
- Podcast Publishing Skill（選用）
- Markdown 輸出 / Notion 整合

### 難度
⭐⭐ 入門

### 預估時間
20–25 分鐘

### 步驟

**步驟 1：安裝影片處理 Skills**
```bash
npx clawhub@latest install youtube-downloader
npx clawhub@latest install whisper-transcriber
npx clawhub@latest install video-summarizer
npx clawhub@latest install podcast-publisher
```

**步驟 2：設定 YouTube 影片抓取**
輸入 YouTube 影片 URL 或頻道 URL。Agent 會自動下載音軌（不需要下載整個影片，只抓音訊可省 80% 空間和時間）。

**步驟 3：執行語音轉文字**
用 Whisper 將音軌轉成逐字稿。設定語言（如果是英文影片想要中文摘要，先轉英文再翻譯效果更好）。開啟時間戳記功能。

**步驟 4：配置摘要模板**
設定你要的摘要格式：
- 三句話摘要（給忙碌的人）
- 重點條列（5–10 個 bullet points）
- 完整章節摘要（按影片段落分段）
- 金句擷取（有價值的引言）

**步驟 5：設定批次處理**
如果你要定期處理某個頻道的影片，設定自動監控：每當頻道發布新影片，自動觸發轉錄 + 摘要流程。

**步驟 6：（選用）產出 Podcast 內容**
將逐字稿改寫成適合 Podcast 的口語化腳本。加入開場白、轉場語、結語。配上背景音樂和 intro/outro 音效。

**步驟 7：建立內容資料庫**
所有摘要和逐字稿自動存入 Notion Database 或本地 Markdown 檔案，建立可搜尋的影片知識庫。

**步驟 8：分享與發布**
將摘要發布到你的部落格、電子報、或社群媒體。搭配教案 06「一文多平台」流程效果更好。

### 常見錯誤
1. **語言設定錯誤**：中英混雜的影片要用 `--language auto` 讓 Whisper 自動偵測，不要硬設單一語言。
2. **摘要太長**：摘要的重點是「節省時間」。如果摘要本身要讀 10 分鐘，那不如直接看影片。控制在原片 10% 以內。
3. **忽略版權問題**：轉錄和摘要他人影片用於個人學習沒問題，但公開發布前要確認是否侵權。
4. **只摘要不行動**：摘要的目的是幫助決策或學習。看完摘要要問自己「下一步要做什麼？」。

### 升級方向
- 影片知識庫 + RAG：搭配教案 04「第二大腦」，讓所有影片摘要可用自然語言查詢
- 自動 Podcast 發布：產出的 Podcast 音檔自動上傳到 Apple Podcasts、Spotify
- AI 配音：用 TTS 將文字摘要轉成語音版本，方便通勤時聽
- 影片精華剪輯：根據摘要中的重點時間戳，自動剪出 3–5 分鐘的精華短影片

## 教案 11：個人 CRM

### 標題
用 AI 打造你的個人 CRM——不再忘記任何一個重要聯絡人

### 副標
自動記錄每次互動、提醒你該跟進誰、把「人脈經營」從記憶力變成系統力。

### 適合誰
- 認識很多人但常忘記上次聊了什麼的業務、顧問、創業者
- 想系統化經營人脈但覺得 Salesforce 太重的自由工作者
- 需要追蹤客戶跟進狀態但團隊太小沒必要買 CRM 系統的小老闆

### 解決什麼問題
你認識的人越來越多，但每次要聯繫某人時都想不起「上次聊了什麼」「他對什麼有興趣」「我答應他什麼」。傳統 CRM 太重、太複雜、不適合個人。你需要一個輕量但聰明的個人關係管理系統。

### 所需工具
- OpenClaw CLI
- Contact Manager Skill
- Email / Calendar Integration Skill
- Reminder Skill
- Notion / Google Sheet 作為資料庫
- Telegram Bot 作為操作介面

### 難度
⭐⭐⭐ 中等

### 預估時間
35–45 分鐘

### 步驟

**步驟 1：定義你的聯絡人分類**
設定你的人脈類別：客戶（潛在/現有/過去）、合作夥伴、業界朋友、導師/被指導者。每個類別設定不同的跟進頻率。

**步驟 2：安裝 CRM Skills**
```bash
npx clawhub@latest install contact-manager
npx clawhub@latest install email-connector
npx clawhub@latest install calendar-connector
npx clawhub@latest install reminder-bot
```

**步驟 3：建立聯絡人資料庫**
在 Notion 或 Google Sheet 建立結構化資料庫：姓名、公司、職稱、認識管道、上次互動日期、互動紀錄、標籤、跟進頻率。

**步驟 4：匯入現有聯絡人**
從 Google Contacts、LinkedIn 匯出、名片掃描等來源匯入你的現有聯絡人。不用一次匯入所有人，先聚焦最重要的 50 人。

**步驟 5：設定自動互動紀錄**
串接 Email 和 Calendar：每當你和某人通信或開會，自動在該聯絡人的紀錄中新增一筆互動記錄（日期、管道、摘要）。

**步驟 6：配置跟進提醒**
根據聯絡人類別設定提醒規則：
- 重要客戶：每 2 週未互動就提醒
- 合作夥伴：每月提醒
- 業界朋友：每季提醒
提醒推送到 Telegram，附上此人的最近互動摘要。

**步驟 7：建立快速記錄介面**
設定 Telegram Bot 指令：快速新增聯絡人、記錄互動、查詢某人資訊。例如：`/log 王小明 今天聊了 Q2 合作方案` 就能快速記錄。

**步驟 8：設定週報與洞察**
每週一自動產出人脈經營報告：本週互動了幾人、哪些人已超過跟進週期、建議本週優先聯繫的 5 個人。

### 常見錯誤
1. **資料太少**：只存姓名和電話沒用。至少要記錄每次互動的重點，這才是 CRM 的價值。
2. **跟進頻率設太高**：每週提醒你聯繫 100 個人，你會直接忽略所有提醒。先從前 20 重要聯絡人開始。
3. **不更新紀錄**：CRM 最怕的是「建好就不用」。養成互動後立即記錄的習慣（所以快速記錄介面很重要）。
4. **把 CRM 當通訊錄**：CRM 的重點不是「存聯絡方式」，而是「經營關係」。每筆紀錄都要回答：他需要什麼？我能提供什麼？

### 升級方向
- AI 破冰建議：根據對方最近的 LinkedIn 動態或新聞，AI 建議你聯繫時可以聊什麼
- 關係圖譜：視覺化你的人脈網路，發現人脈中的關鍵節點
- 自動跟進信件：AI 根據互動紀錄草擬跟進信件，你確認後自動寄出
- 銷售漏斗追蹤：追蹤潛在客戶從初次接觸到成交的完整旅程

## 教案 12：多代理網站改版

### 標題
6 個 AI 代理同時作業——40 分鐘完成網站改版

### 副標
別再一頁一頁慢慢改。讓 6 個 AI Agent 分工合作：一個改首頁、一個改導航、一個改樣式——全部同時進行。

### 適合誰
- 需要快速改版但前端工程師排不出時間的新創團隊
- 自己經營網站但不想花整個週末改版的個人品牌
- 想體驗多代理（Multi-Agent）協作威力的技術愛好者

### 解決什麼問題
傳統網站改版是線性作業：先改結構、再改樣式、再改內容，一個人做要好幾天。你的網站很多頁面其實可以同時修改——只要有多個 Agent 同時作業，且彼此不會衝突。

### 所需工具
- OpenClaw CLI
- Multi-Agent Orchestrator Skill
- Web Development Skills（HTML/CSS/JS 修改）
- Git Version Control（確保多 Agent 不衝突）
- 設計稿或改版規格說明（你要改成什麼樣子）

### 難度
⭐⭐⭐⭐ 進階（需要理解 Git 分支和多代理協作概念）

### 預估時間
40–60 分鐘

### 步驟

**步驟 1：撰寫改版規格文件**
明確定義你要改什麼：哪些頁面要改、色彩方案、版型調整、新增/刪除哪些區塊。越具體越好。AI Agent 需要清楚的指令才能正確執行。

**步驟 2：安裝多代理 Skills**
```bash
npx clawhub@latest install multi-agent-orchestrator
npx clawhub@latest install web-dev-agent
npx clawhub@latest install css-styler
npx clawhub@latest install git-manager
```

**步驟 3：規劃 Agent 分工**
將改版任務拆分成互不衝突的工作單元，分配給不同 Agent：
- Agent 1：首頁 Hero 區塊 + CTA
- Agent 2：導航列 + 頁尾
- Agent 3：全站色彩與字型樣式
- Agent 4：案例頁面內容更新
- Agent 5：響應式設計調整
- Agent 6：SEO meta tags + Open Graph

**步驟 4：設定 Git 分支策略**
每個 Agent 在自己的 Git 分支上作業。完成後合併到 staging 分支。設定自動衝突偵測——如果兩個 Agent 修改了同一個檔案，暫停並通知你手動處理。

**步驟 5：啟動多代理作業**
同時啟動所有 Agent。監控各 Agent 的進度。Orchestrator 會協調它們的作業順序，確保有依賴關係的任務按正確順序執行。

**步驟 6：即時品質檢查**
每個 Agent 完成後自動執行檢查：HTML 驗證、CSS 語法檢查、連結是否正常、RWD 是否破版。有問題的自動退回修正。

**步驟 7：合併與整合測試**
將所有分支合併到 staging。執行完整的跨頁面測試：導航是否正常、樣式是否一致、表單是否能提交。

**步驟 8：預覽確認與上線**
在 staging 環境完整預覽改版成果。確認沒問題後合併到 main 分支並部署。保留舊版備份以備回滾。

### 常見錯誤
1. **任務拆分有重疊**：兩個 Agent 都去改 `styles.css` 會產生衝突。任務拆分時要確保每個 Agent 負責不同的檔案或區塊。
2. **規格太模糊**：「讓網站更好看」不是好的規格。要具體到「Hero 區塊背景改為漸層色 #6ea8ff → #8c6bff」。
3. **沒有 staging 環境**：直接在 production 上跑多 Agent 改版是災難。一定要有 staging 環境。
4. **忽略行動版**：桌面版改好了但行動版破版。一定要在步驟 3 就把 RWD 調整列為獨立任務。

### 升級方向
- 設計稿自動轉程式碼：上傳 Figma 設計稿，AI 自動產出對應的 HTML/CSS
- A/B 測試部署：同時部署兩個版本，自動追蹤轉換率差異
- 持續改版流程：不再是「大改版」，而是每週小幅持續優化
- 多語系網站改版：同步處理中文、英文、日文版本
